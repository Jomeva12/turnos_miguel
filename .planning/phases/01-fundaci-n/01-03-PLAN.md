    1. Create `lib/actions/employees.ts` with 'use server' directive:

    Export async function `toggleEmployeeArea(employeeId: number, areaId: number, enabled: boolean)`:
       - Check session via `auth.api.getSession({ headers: await headers() })` — import headers from "next/headers"
       - If no session, throw new Error("No autorizado")
       - If enabled: upsert EmployeeArea (create if not exists, no-op update if exists)
       - If not enabled: deleteMany EmployeeArea where employeeId AND areaId match
       - Call `revalidatePath("/habilidades")` after mutation
       - Return nothing (void) — the revalidation handles UI refresh

    2. Use the Prisma singleton from `@/lib/db`, NOT a new PrismaClient.

    3. Use upsert with composite key: `where: { employeeId_areaId: { employeeId, areaId } }` for the create case. Use deleteMany for the delete case (safe even if record doesn't exist).

    4. Create `lib/actions/employees.test.ts` to verify toggleEmployeeArea behavior:
       - Import PrismaClient

         - "should be idempotent on create" — toggle true twice, verify only one record exists
         - "should be idempotent on delete" — toggle false on non-existent record, verify no error
       - Use beforeAll/afterAll for PrismaClient lifecycle
       - Note: Since Server Actions require HTTP context, these tests validate the Prisma operations directly (upsert/deleteMany patterns), not the full action with auth check
